# Proposal: v76 Mora Temprana (Early Delinquency Visibility)

## Intent

The backend delinquency evaluation engine successfully calculates arrears and updates account statuses to "Delinquent" with appropriate aging buckets in the background. However, operators have no way to view this data. This proposal aims to unblock the collections roadmap by surfacing the results of the existing backend worker through a read-only API and a corresponding frontend UI. 

## Scope

### In Scope
- Create read-only HTTP endpoints (`DelinquencyController`) to query delinquency records.
- Implement MediatR queries (`GetDelinquentAccountsQuery`) for the existing `DelinquencyRecordEntity`.
- Establish authorization policies (e.g., `CanViewCollections`) to restrict access.
- Add an Angular feature module for collections with a `delinquency-list` component.
- Display delinquent accounts with their aging buckets (1-30, 31-60, 61-90, >90 days).

### Out of Scope
- Adding mutation actions for collections (registering contact, tracking outcomes) - deferred to v77+.
- Modifying the existing background worker logic (`DelinquencyEvaluationWorker`).
- Modifying the core ledger or billing logic.

## Capabilities

### New Capabilities
- `delinquency-management`: Provides visibility and tracking of accounts in arrears and their respective aging buckets.

### Modified Capabilities
- `identity-and-access`: Add permissions/policies to authorize collections visibility.

## Approach

We will build the "smallest sensible first slice" by solely surfacing the data generated by the existing CQRS background worker. This avoids scope creep and allows validation of the worker's data before adding complex mutation logic. The frontend will get a new collections module since this is a distinct banking domain, while the backend API will expose paginated results based on the `DelinquencyRecordEntity` read model.

## Affected Areas

| Area | Impact | Description |
|------|--------|-------------|
| `backend/services/CardVault/src/CardVault.Api/Controllers` | New | Add `DelinquencyController` |
| `backend/services/CardVault/src/CardVault.Application/Delinquency` | New | Add MediatR queries and dtos |
| `frontend/src/app/features/collections` | New | Add Angular module and `delinquency-list` component |
| `backend/services/CardVault/src/CardVault.Api/Authorization` | Modified | Add `CanViewCollections` policy |

## Risks

| Risk | Likelihood | Mitigation |
|------|------------|------------|
| Unauthorized access to sensitive collection data | High | Implement and test strict RBAC authorization policies before exposing endpoints. |
| Inconsistent pagination/filtering between API and UI | Medium | Standardize query contracts following existing `http-contracts` conventions. |

## Rollback Plan

- **API:** Remove or disable `DelinquencyController` and its associated routes.
- **Frontend:** Remove the `collections` route module from the Angular routing configuration.
- **Auth:** Revert policy additions in the Identity configuration.

## Dependencies

- Existing background worker data generation (`DelinquencyRecordEntity` population).

## Success Criteria

- [ ] A user with `CanViewCollections` permission can view a paginated list of accounts in arrears.
- [ ] Accounts in the UI correctly display their respective aging buckets (e.g., 1-30, 31-60 days).
- [ ] A user without the required permission receives a 403 Forbidden response.
