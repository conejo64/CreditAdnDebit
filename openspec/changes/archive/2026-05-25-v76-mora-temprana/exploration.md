## Exploration: v76-mora-temprana

### Current State
For `v76` (Gestin de mora temprana), the backend has successfully implemented the automated background processes using CQRS:
- `DelinquencyEvaluationWorker.cs`: A background service that runs daily.
- `EvaluateDelinquencyCommandHandler.cs`: Calculates days in arrears, updates the `Account.Status` to `Delinquent`, and creates/updates `DelinquencyRecordEntity` with aging buckets (1-30, 31-60, 61-90, >90 days).
- The database schema (Migrations, `DelinquencyRecordEntity`) is complete.
However, there is absolutely no way to visualize this data. No API endpoints expose the delinquencies, and the frontend lacks the corresponding components.

### Affected Areas
- **Backend API**: Missing a `DelinquencyController` and MediatR queries to read the generated delinquency records.
- **Frontend UI**: Missing an Angular feature module (likely a new `collections` or within `finance`) to display the delinquent accounts and their aging buckets.
- **OpenSpec**: The `openspec/specs` directory has no documentation for delinquency or collections contracts.

### Approaches

1. **Complete v76 as a Read-Only Slice (Recommended)**
   - **Backend**: Implement `GetDelinquentAccountsQuery` and a `DelinquencyController` to expose the daily batch results. Add an authorization policy (e.g., `CanViewCollections`).
   - **Frontend**: Create a `delinquency-list` component in Angular to view the accounts in arrears and their respective buckets.
   - **Pros**: Smallest sensible slice. Validates the backend worker's results visually without overcomplicating the scope. Closes the gap defined in `Sprints.md` ("Falta Front").
   - **Cons**: Read-only; operators cannot register actions yet.
   - **Effort**: Low

2. **Combine v76 API with v77 (Bandeja de Cobranzas)**
   - Build the API for v76 and immediately add the mutation actions from v77 (register contact, track outcomes).
   - **Pros**: Delivers a fully actionable collections inbox at once.
   - **Cons**: High risk of scope creep. Introduces new domain entities (contact logs) before the base v76 view is verified.
   - **Effort**: High

### Recommendation
Proceed with **Approach 1**. 
We should propose a change that purely surfaces the data generated by the existing `DelinquencyEvaluationWorker`. This requires a new API controller, corresponding read models/queries, and the frontend UI scaffold that was noted as missing in `Sprints.md`. This is the perfect "smallest sensible first slice" to unblock the collections roadmap.

### Risks
- **Authorization**: A new security policy (e.g., `CanViewCollections`) and role assignments will be needed, as delinquency data is sensitive.
- **Frontend Structure**: We need to decide if this lives under `features/finance` or a new `features/collections` domain in the Angular app. Given the banking context, a separate `collections` module is standard.
- **Specifications**: Since no specs exist for collections, we'll need to establish the baseline `spec.md` for this domain.

### Ready for Proposal
Yes. The orchestrator can proceed to the propose phase based on Approach 1.
