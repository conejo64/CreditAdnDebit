# Code Review 审查报告生成技能

## 触发条件
用户要求对代码、PR、diff 或文件变更进行代码评审，并希望得到书面/Markdown 格式的审查报告。

## 工作流程
1.  **前置校验**
    - 明确评审范围：文件路径、Git diff、提交记录或代码片段
    - 确认评审目标：功能正确性、性能、安全、规范、可维护性
    - 识别项目技术栈与编码规范（如 ESLint/Prettier/TS 规则）

2.  **多维度审查**
    - ✅ **规范合规性**：命名、格式、注释、文件结构、工程约定
    - ✅ **功能正确性**：逻辑完整性、边界条件、异常处理、业务规则匹配
    - ✅ **性能优化**：循环冗余、重复请求、N+1 查询、资源泄漏、并发隐患
    - ✅ **安全审计**：权限控制、注入风险、敏感信息、越权操作、参数校验
    - ✅ **可维护性**：耦合度、职责单一、魔法值、硬编码、测试覆盖
    - ✅ **架构一致性**：依赖管理、模块划分、设计模式使用、扩展性

3.  **报告生成规则**
    - 按 `严重/警告/提示` 三级分级，标注问题位置（文件+行号）
    - 每个问题包含：问题描述、风险影响、整改建议、代码示例（修正前后对比）
    - 末尾生成 **可执行的整改清单**（按优先级排序）与 **整体质量评估**
    - 输出为标准 Markdown 格式，支持直接用于团队评审或 Issue

## 输出模板
```markdown
# Code Review Report

## 基本信息
- 评审对象：`{文件/PR/diff 路径}`
- 评审范围：`{变更简述}`
- 评审时间：`{时间}`
- 技术栈：`{语言/框架}`

## 问题汇总
| 级别 | 文件位置 | 问题描述 | 风险影响 | 整改建议 |
|------|----------|----------|----------|----------|
| 严重 | `src/xxx.ts:45` | 未校验用户输入直接拼接 SQL | SQL 注入风险 | 使用参数化查询，示例：`const sql = 'SELECT * FROM users WHERE id = ?';` |
| 警告 | `src/yyy.ts:12` | 重复请求同一接口 3 次 | 性能浪费，高并发下易超时 | 缓存结果或合并请求，示例：`const data = await fetchData();` |

## 整改优先级清单
1.  高优先级：修复所有严重级问题（安全/数据风险）
2.  中优先级：处理警告级问题（性能/可维护性）
3.  低优先级：优化提示级问题（规范/风格）

## 整体评估
- 合规性：{评分/结论}
- 性能：{评分/结论}
- 安全性：{评分/结论}
- 可维护性：{评分/结论}
```
